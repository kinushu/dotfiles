# memory-dream

## これは何か / なぜ必要か

エージェントはセッションごとに記憶へ **局所的・増分的に追記** する。1 セッションの追記はその場では正しくても、追記だけを続けると記憶全体は次第にノイズ化する。具体的には次のような劣化が積み重なる。

- 同じ知見が複数箇所に重複して書かれる（どれが正か分からなくなる）。
- 後から得た知見が古い記述と矛盾する（両方が残り、どちらを信じるべきか判断できない）。
- 「昨日」「先週」といった相対日付が、読み返す時点では意味を失う。
- すでに削除・改名されたファイルや廃止された手順を指す記述が残り、誤誘導する。

こうなると記憶は「思い出す助け」ではなく「混乱させるノイズ」に転落する。`memory-dream` は、追記で溜まった記憶を定期的に **再編（consolidation）** し、重複・矛盾・陳腐化を除去して「今この瞬間に正しい知見だけが、一箇所に、絶対的な表現で」残る状態へ戻す playbook である。睡眠中に記憶を整理する Dreams のメタファに由来する。

**大原則:**

- **入力非破壊**: 元の記憶を直接上書きで壊さない。再編結果は別途生成し、元をいつでも復元できる状態を保つ。
- **採用前レビュー必須**: consolidation の出力には要約・言い換えの過程で hallucination（実在しない知見の捏造や、原文の意味の取り違え）が混入しうる。生成結果をそのまま自動採用してはならず、必ず人間のレビューを経てから採用する。
- **consolidation は fine-tune ではない**: これはモデルの重みを更新する学習ではなく、あくまで「テキストとしての記憶ファイルを整理し直す」編集作業である。整理によって新しい能力が生まれるわけではない。

## 4 フェーズ手順

記憶の再編は次の 4 フェーズを順に実施する。

### 1. Mine（採掘）

再編の材料を集める。

- 既存の記憶ファイル群と索引を読み、現在の記憶の全体像を把握する。
- 利用可能であればセッションの記録（対話ログ等）から、まだ記憶化されていない有用な知見・確定した事実・再現可能な手順を採掘する。
- この段階では取捨選択せず、候補を広く拾う。採用判断は後のフェーズで行う。

### 2. Consolidate（統合）

採掘した素材と既存記憶を、意味のあるまとまりへ統合する。

- 同一トピックの断片を 1 つのまとまった記述に束ねる。
- 断片的な観察を、一般化できる知見・ルール・手順へ昇華する。
- 冗長な言い回しを削り、簡潔で読みやすい形に整える。

### 3. Dedup & Resolve（重複排除・矛盾解消）

重複を排除し、矛盾を解消する（判定ルールは後述）。

- 同じ内容が複数箇所にある場合、正しい配置を 1 箇所だけに決めて残りを削除する。
- 互いに矛盾する記述は、最新かつ正しい値に一本化する。
- どちらが正か判断できない矛盾は、勝手に決めず保留してレビュー時に確認する。

### 4. Prune & Index（剪定・索引化）

不要になった記述を剪定し、索引を整える。

- 陳腐化した記述（存在しない参照、廃止された手順、無意味になった相対日付）を除去する。
- 相対日付を絶対日付へ変換する。
- 索引を lean（無駄なく最小限）に保つ。索引は「何がどこにあるか」を素早く引くためのもので、本文の要約を肥大させない。

## 重複排除・矛盾解消の判定ルール

- **下位 → 上位方向で修正する**: より局所的・具体的なレイヤ（個別の記憶ファイル等）から、より広く共有されるレイヤ（上位ルール・索引等）へ向かって整理する。局所の知見が一般則へ昇格できるなら上位へ移し、局所側からは消す。
- **定義は一箇所（single source）**: 同じ定義・ルール・手順は必ず 1 箇所にだけ置く。参照が必要なら「どこにあるか」を指すに留め、内容そのものを二重に書かない。
- **上位レイヤが定めるルールを下位で再掲しない**: 上位で既に定義済みのルールを下位の記憶ファイルへコピーしない。上位が更新されたとき下位が古いまま取り残され矛盾の温床になるためである。
- **重複回避の注記自体も成果物に残さない**: 「これは上位に定義済みなので省略」「重複を避けるためここでは書かない」といったメタな断り書きも、最終成果物にはノイズとして残さない。整理した結果だけを残す。
- **矛盾は最新かつ正しい値に一本化する**: 時間とともに更新された事実は、最新の正しい値のみを残す。古い値は履歴として残さず削除する。判断がつかない矛盾は保留し、採用前レビューで確認する。

## 成果ファイルの書き方

再編後の記憶ファイルには、**現時点で正しい知見・ルール・再現手順だけ** を残す。以下は成果物から除去する。

- 経緯・履歴（どういう順序でこの結論に至ったか）
- Why の物語や意思決定の背景説明（現行の結論だけあれば足りる）
- 失敗談・試行錯誤の記録
- 「◯回目の再発」といった再発回数のカウント
- 学習日・記録日・セッション ID などのトレーサビリティ用メタデータ

一方で、**技術的な因果関係（「A をすると B が壊れるので C する」等、再現・判断に必要な因果）は残してよい**。除くのは「誰が・いつ・何回目に気づいたか」という属人的・時系列的な文脈であって、知見そのものの因果構造ではない。

原則として「その記述を読んだ人が、今すぐ正しく行動できるか」だけを基準に取捨する。行動に不要な物語は残さない。

## 相対日付 → 絶対日付変換

「昨日」「先週」「最近」などの相対表現は、読み返す時点で意味を失うため、記憶化の際に **絶対日付（YYYY-MM-DD 等）へ変換** する。日付そのものが知見の一部として不要なら、日付自体を削る（前節の「学習日を残さない」と整合）。

## 索引を lean に保つ

索引は本文への入り口であり、本文の複製ではない。1 エントリにつき最小限のフック（何がどこにあるかを引ける 1 行程度）に留め、索引が肥大して本文と二重管理にならないようにする。

## トリガー条件

以下のいずれかで実行を検討する。

- 大規模リファクタや構成変更の直後（既存記憶中の参照・手順が一斉に陳腐化しやすい）。
- セッションが一定数蓄積したとき（目安 20〜30 セッション。追記が溜まり重複・矛盾が生じ始める）。
- ユーザーが「記憶の整理」「dream」「memory consolidation」等を明示的に指示したとき。

## 採用前レビュー

consolidation の出力は自動採用しない。要約・統合の過程で hallucination が混入しうるため、必ず人間が生成結果をレビューし、承認を得てから採用する。元の記憶は非破壊で保持し、レビューの結果いつでも元へ戻せる状態を保つ。

## チェックリスト

再編を採用する前に、以下を確認する。

- [ ] 相対日付を絶対日付へ変換した（または不要な日付を削除した）
- [ ] 上位レイヤで定義済みのルールを下位の記憶から削除した
- [ ] 経緯・履歴・Why の物語・再発回数・学習日・セッション ID を成果物から除いた
- [ ] 矛盾を最新かつ正しい値へ一本化した（判断不能なものは保留した）
- [ ] 存在しないファイル・廃止手順への参照を除去した
- [ ] 同一の定義・ルールが一箇所だけに存在する（重複が無い）
- [ ] 重複回避の断り書き等のメタ注記を残していない
- [ ] 索引が lean（最小限のフックのみ）に保たれている
- [ ] 元の記憶を非破壊で保持し、採用前レビューを実施した
