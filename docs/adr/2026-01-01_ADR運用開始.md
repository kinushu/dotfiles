# ADR運用開始

## Context and Problem Statement

dotfilesプロジェクトにおいて、アーキテクチャ上の決定事項を記録・追跡する仕組みがなかった。設計判断の経緯や理由が暗黙知となり、将来のメンテナンスや見直し時に困難が生じる可能性がある。

## Decision Drivers

* 設計判断の理由を将来の自分が振り返れるようにしたい
* 決定事項の変更履歴を追跡可能にしたい
* Claude Codeとの協働時に判断基準を共有したい

## Considered Options

1. ADRをMADR形式で運用
2. READMEに決定事項を追記
3. Wikiで管理
4. 運用しない（現状維持）

## Decision Outcome

選択: ADRをMADR形式で運用

理由:
- MADR形式は軽量で導入しやすい
- Markdown形式でリポジトリ内に保存でき、バージョン管理と一体化できる
- Claude Codeのスキルと連携し、ADR作成を自動化できる

## Consequences

### Good

* 設計判断の経緯が明文化され、振り返りが容易になる
* 新しい決定を下す際に過去の判断を参照できる
* AIエージェントとの協働時にコンテキストを共有できる

### Bad

* ADR作成の手間が増える（スキルで軽減）
* 運用が形骸化する可能性がある

## More Information

### 運用ルール

1. 重要な設計判断を行う際にADRを作成する
2. `/create_adr` スキルまたは `ADR作成` と指示してClaude Codeに作成を依頼
3. ステータスは `proposed` → `accepted` / `deprecated` / `superseded` の順で更新
4. ファイル名は `YYYY-MM-DD_タイトル.md` 形式
