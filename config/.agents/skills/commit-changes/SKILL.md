# コミットメッセージ作成とコミット実行

## 目的

- 変更差分を確認し、意図が分かる日本語コミットメッセージを作成する
- 先頭に必ず特性プリフィックス（`[fix]` など）を付ける
- メッセージ作成だけでなく `git commit` 実行までを一貫して行う

## プリフィックス一覧と判断基準

- `[fix]`: バグ修正、不具合回避、誤動作の修正
- `[add]`: 新機能・新規設定・新規ファイル追加
- `[update]`: 既存機能や既存設定の挙動更新、改善
- `[remove]`: 機能・設定・ファイルの削除
- `[docs]`: ドキュメントのみの変更
- `[chore]`: ビルド・CI・依存更新・雑多な保守作業

迷う場合は、変更の主目的に最も近いものを1つ選ぶ。

## 手順

1. `git diff --staged` でステージ済み変更を確認する
2. 変更の主目的を判定し、プリフィックスを1つ選ぶ
3. 以下の形式でコミットメッセージを作成する
   - 1行目: `[prefix]` + 半角スペース + 変更要約（50文字以内推奨）
   - 空行
   - 詳細説明（なぜこの変更が必要か）
4. 生成元情報（`Co-Authored-By` など）は本文から除去する
5. 次の形式でコミットを実行する
   - `git commit -m "[fix] 変更要約" -m "詳細説明..."`

## フォーマット例

```text
[add] ユーザー認証にOAuthログインを追加

GoogleとGitHubアカウントでのログインを可能にした。
既存のメール認証との併用も可能にして移行コストを下げた。
```

```bash
git commit -m "[add] ユーザー認証にOAuthログインを追加" -m "GoogleとGitHubアカウントでのログインを可能にした。既存のメール認証との併用も可能にして移行コストを下げた。"
```

## 移行案内

- 旧スキル名 `commit-message` は移行用エイリアスとして維持する
- 新規利用は `commit-changes` を使用する
- `commit-message` が呼ばれた場合も、本スキルと同じ手順でコミット実行まで行う
