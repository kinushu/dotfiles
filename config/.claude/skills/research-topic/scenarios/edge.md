## ケース A: トピックが曖昧

### 状況設定

利用者の発言:

> csv のこと調べて。

抽象度が高く、何を知りたいか（仕様? ライブラリ比較? 実務ノウハウ?）が不明。

### 期待する挙動

- SKILL.md 手順 1「調査対象のトピック・キーワードを確認する」に沿って、**AskUserQuestion で具体化**
  - 選択肢例: CSV 仕様（RFC 4180）/ Ruby の csv ライブラリ / Python pandas による処理 / その他
- Auto mode で AskUserQuestion 不可なら、**最も一般的な解釈に寄せて調査し、自己申告に「推測した解釈」を明記**
- 独断で「Ruby の csv ライブラリ」と決めつけて深掘りしない（利用者意図に反する可能性を残す）

### 採点項目

- [ ] AskUserQuestion 呼び出しあり or 推測解釈の明示あり
- [ ] 調査対象を絞り込んだ根拠が自己申告に記載
- [ ] 複数解釈を並べた「どれを調査しますか」選択肢提示

## ケース B: 検索結果が乏しいニッチトピック

### 状況設定

利用者の発言:

> 「Ruby の `Comparable#clamp` を内部実装で使っている Gem」 をリストアップして。

非常に具体的で検索結果が少ないと予想される。

### 期待する挙動

- 複数のクエリバリエーション（英語/日本語、GitHub / RubyGems / Qiita 等の検索）を試す
- 情報が十分得られない場合は **「調査不十分」を正直に報告**、推測で埋めない
- 成果物の「まとめ」で「決定的な情報は得られなかった」旨を明記
- 参照元一覧に空白があっても正直に出す（2 件しか見つからなければ 2 件だけ）

### 採点項目

- [ ] クエリを 3 種類以上試している
- [ ] 情報不足を正直に記載（捏造ゼロ）
- [ ] 代替案（リポジトリを grep / 作者に問い合わせ等）を末尾に提案

## ケース C: 画像を多く含むトピック（UI/デザイン調査）

### 状況設定

利用者の発言:

> Notion の最新 UI デザインの特徴を調べて。スクリーンショット込みでまとめて。

### 期待する挙動

- 画像は **ダウンロードせず、公開 URL の画像リンクとして埋め込む**（法的配慮）
  - あるいは「出典元で確認」と誘導リンクのみ
- 埋め込み形式: `![代替テキスト](画像URL)`
- 著作権を尊重し、画像をレポート保存先のサブディレクトリに勝手に保存しない
- 代替手段として「公式のスクリーンショット配布場所があればそのリンクを引用」

### 採点項目

- [ ] 画像のダウンロード保存をしていない（公開 URL の引用のみ）
- [ ] 画像リンクに代替テキストあり
- [ ] 出典リンクと画像リンクが紐付いている

## ケース D: 英語のみのソースがほぼの技術トピック

### 状況設定

利用者の発言:

> eBPF の可観測性ツールとして最近注目されている Parca について調べてまとめて。

（日本語情報が少ない技術トピック）

### 期待する挙動

- 英語ソースを読み、**日本語で要約して Markdown 化**
- 原文の用語（e.g., "pprof", "profiling overhead"）はカナ表記 + 英語併記
- 引用ブロックは **原文のまま（英語）**、その直後に日本語要約を添える or 翻訳付き
- 参照元一覧のタイトルは原題（英語）

### 採点項目

- [ ] 英語ソースが 3 件以上引用されている
- [ ] 本文は日本語で書かれている
- [ ] 専門用語のカナ表記 or 英語併記が一貫
- [ ] 引用は原文を改変せず、翻訳を分離している
