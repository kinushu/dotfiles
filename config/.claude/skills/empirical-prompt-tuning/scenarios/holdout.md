# scenario: holdout

**役割**: 書き手が対策を打っていない未見ケース。過学習（書き手が median / edge で見えた問題にだけピンポイント対処して SKILL.md を書き換える現象）の検出用。

**運用ルール（重要）**:

- このファイルは **評価の最終回まで開かない**。
- SKILL.md を改訂している途中で中身を参照すると、対策が混入してしまい検出能力を失う。
- 収束判定のタイミング（フェーズ 7）で初めて開いて subagent に dispatch する。
- 一度使ったら、次のチューニングサイクルでは **別の未見ケースに差し替える**（使い回すと hold-out でなくなる）。

## Output 採点項目

- [ ] 改訂後も median / edge と同等のスコア
- [ ] 改訂前と比べて劣化していない
- [ ] 書き手が想定していなかったはずの分岐でも破綻しない

## 劣化検出の指標

- tool_uses が median の 1.5 倍以上に膨らむ → 過学習の疑い
- Output 採点で 1 項目でも Fail → 収束とみなさず差分修正へ戻る
- subagent の自己申告で「SKILL.md のどの手順を適用すべきか判然としない」と申告 → 汎化不足

## 更新ポリシー

- holdout は **使い捨て**。1 回のチューニングサイクルで消費する。
- 次回以降に備えて、対象 skill の改訂が決まったタイミングで新しい holdout を仕込んでおく。
- 仕込む担当は「その skill の改訂者とは別の人物 or 同一人物でも 1 週間以上空けた自分」が理想。
