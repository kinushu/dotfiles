# scenario: median — 技術トピックの調査

**役割**: 特定の技術トピックを複数ソースから調査し、Markdown レポートにまとめる典型用途。

## 前提状況

- トピック: **「Ruby 3.4 で追加された新機能」**（2024 年末〜2025 年にかけてリリースされた実在技術）
- 利用者の用途: 自分の開発環境アップデート検討のための情報集め
- 出力先希望: `tmp/` 配下（既定想定）
- 期限感: 即座に読める品質。後日の再調査にも耐える構造

## 入力

利用者の発言（想定）:

> Ruby 3.4 の新機能について調べてまとめて。YJIT 改善とかパフォーマンス面が気になる。Prism パーサーの話も知りたい。

## 期待する成果物

- ファイル: `tmp/2026-04-23_Ruby_3.4_新機能_research.md`（日付は実行日）
- 中身: SKILL.md のテンプレート準拠
  - `# 調査レポート: Ruby 3.4 の新機能`
  - `調査日: 2026-04-23`
  - `## 概要`
  - `## 再調査時のプロンプト案`
  - `## まとめ`
  - `## 調査結果` 配下にサブトピック（例: YJIT / Prism / それ以外の主要変更 / マイグレーション注意点）
  - 各サブトピックに出典付き引用ブロック
  - `## 参照元一覧` 表（タイトル・URL・参照日）

## Output 採点項目

- [ ] テンプレートの全セクションが埋まっている（概要 / 再調査プロンプト / まとめ / 調査結果 / 参照元一覧）
- [ ] サブトピックが 3 つ以上、論理的に整理されている
- [ ] 各サブトピックに少なくとも 1 件の出典付き引用
- [ ] 参照元一覧の行数 ≥ サブトピック数（ユニーク URL ベース）
- [ ] 引用フォーマット `> 引用\n>\n> — 出典: [タイトル](URL)` が崩れていない
- [ ] 同じ URL を複数回引用する場合、無駄な重複がない
- [ ] 執筆日 `YYYY-MM-DD` が記載されている

## Process 採点項目

- 想定 tool_uses 数: 8〜20（Read skill + WebSearch 数回 + WebFetch 数回 + Write + md-open）
- 必須 tool:
  - SKILL.md を Read
  - WebSearch / WebFetch で情報収集（少なくとも 3 ソース）
  - Write（レポートファイル）
  - Bash (`~/bin/md-open` 最終版 1 回)
- 迂回チェック:
  - 出力先を勝手に決めず AskUserQuestion（Auto mode なら既定値でフォールバックして自己申告）
  - 検索クエリが冗長/重複していないか
  - 画像が無いトピックで無理に画像リンクを埋めていないか

## 注意

- subagent には「Auto mode で AskUserQuestion 不可」と prompt で明示し、出力先既定値（例: `tmp/`）を与える
- 実ファイル作成は subagent の返答内容として ADR 本文を提示する形にする（実環境汚染を避ける）
- md-open 実行指示は省略してよい（ファイル未作成のため）
