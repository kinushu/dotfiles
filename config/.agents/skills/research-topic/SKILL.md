# トピック調査スキル

指定されたトピック・キーワードについて Web 検索などで調査し、結果を Markdown ファイルにまとめる。

## 実行手順

1. **調査対象の明確化** — トピック・キーワード・深掘り観点を確認
2. **出力先の決定** — AskUserQuestion で確認（Auto mode 時は既定値にフォールバック）
3. **情報収集** — WebSearch / WebFetch で複数ソースから収集
4. **Markdown レポート作成** — 後述のテンプレートに従って作成
5. **プレビュー** — 最終成果物として提示する時点で `~/bin/md-open` を 1 回のみ実行

## 1. 調査対象の明確化

利用者入力から以下を読み取る。不明なら AskUserQuestion で補う。

- **対象トピック**（名詞。「〜について」の主語）
- **深掘り観点**（何を知りたいか。例: 性能 / 導入手順 / 比較）
- **スコープ絞り込み**（バージョン、言語、業界 etc.）
- **時期感**（最新のみ / 歴史も含めて）

曖昧なトピック（例: 「csv のこと調べて」）の場合は、複数解釈を提示して利用者に選ばせる。**独断で解釈を狭めない**。Auto mode で AskUserQuestion 不可の場合は、最も一般的な解釈に寄せ、自己申告として推測内容を明記する。

## 2. 出力先の決定

ファイル出力前に AskUserQuestion で以下を確認する。

### .md ファイル
- 出力ディレクトリ（既定の優先順位: `tmp/` → `docs/research/` → カレントディレクトリ）
- トピックファイル名 `{YYYY-MM-DD}_{HHMM}_{トピック名}_research`
  - `{YYYY-MM-DD}` は `date +%Y-%m-%d`、`{HHMM}` は `date +%H%M`
  - トピック名は基本的に日本語、英語、半角数字、`_` で構成。空白や記号は `_` に置換
- ファイル名は `{トピックファイル名}.md`

### 関連画像・PDF
- 出力ディレクトリ内に `{トピックファイル名}`（拡張子 `.md` を除いた名前）のサブディレクトリを作成
  - 例: `tmp/2026-04-23_ruby_3_4_新機能_research/image1.png`
- 画像の扱いは「4-1. 画像引用ポリシー」に従う

### Auto mode / subagent でのフォールバック

AskUserQuestion が使えない環境では、以下の既定値で進めて **自己申告に明記**:

- 出力ディレクトリ: `tmp/`
- ファイル名: `{YYYY-MM-DD}_{HHMM}_{トピック名}_research.md`
- 画像保存: **行わない**（外部リンク引用のみ。下記ポリシー参照）

## 3. 情報収集

### 検索戦略

- **WebSearch クエリ回数**: 2〜4 回を目安。初回で広く、2 回目以降で深掘りする
- **WebFetch URL 件数**: 3〜6 件を目安。信頼度順に選ぶ
- **合計 Web 呼び出し**: 10 回以内が原則（コスト抑制）

情報源は **信頼度の高い順** に参照する:

1. 公式（プロジェクト公式サイト、公式 GitHub、公式リリースノート）
2. 一次資料（論文、仕様書、RFC、公式ブログ）
3. 実測データ（ベンチマーク記事、GitHub issue の再現確認）
4. 評価の確立した技術ブログ（執筆者が特定でき、出典が付いている）
5. まとめサイト（最後の手段、必ず原文ソースも確認）

### 複数ソース必須

- **最低 3 ソース** を参照（同ソースの複数ページは 1 ソースとカウント）
- 英語/日本語どちらのソースでも可。日本語情報が乏しい技術は英語優先で日本語要約
- 意見の分かれるトピックは、**対立する立場のソースを両方引用** する

### 検索で十分な情報が得られない場合

- クエリを 3 種類以上試して結果が乏しい場合、「調査不十分」を正直に報告
- 推測で情報を補わない（**捏造禁止**）
- 「まとめ」節に「決定的な情報は得られなかった」旨を明記し、代替手段（原文リポジトリを grep、作者に直接確認 等）を提案

## 4. Markdown レポート作成

後述の「出力形式」に従う。調査中は以下をメモリ上で積み上げていく。

- 引用した原文スニペット（コピー）
- 出典のタイトル / URL / アクセス日
- サブトピックの論理的整理（重複除去、階層化）

Write ツールで確定版を書き出す（**段階的な中間保存は不要**、最終版 1 回で十分）。

### 4-1. 画像引用ポリシー

- **外部画像の埋め込み**: 公開 URL の画像リンクとして引用する（`![代替テキスト](画像URL)`）
- **画像のダウンロード保存**: 原則行わない（著作権の配慮）。ただし利用者が明示的に許可した場合、または公式がダウンロード配布している素材の場合のみ許可
- 代替テキストは必ず埋める（画像が出ないとき何の画像だったかが分かるように）
- 画像ソースが特定できない場合は埋め込まず、出典リンクに誘導する文章で代替

### 4-2. 先頭トピックアイコン

レポートの **タイトル直下（調査日より前）** に、トピックを象徴するアイコン・シンボル画像を 1 枚埋め込む。

- **画像の選定**: トピックを最も端的に表す公式ロゴ・アイコン・シンボルを優先する（例: 言語/ツールの公式ロゴ、規格のシンボルマーク）
- **取得方法**: 公式サイト・公式 GitHub・Wikipedia/Wikimedia Commons など出典が明確な公開 URL を使う。画像 URL は調査中の WebSearch / WebFetch で見つけたものを使い、リンク切れに注意する
- **埋め込み形式**: 「4-1. 画像引用ポリシー」に準拠し、外部リンク埋め込み（`![代替テキスト](画像URL)`）とする。ダウンロード保存はしない
- **代替テキスト**: トピック名を含めて必ず埋める（例: `![Ruby ロゴ](...)`）
- **見つからない場合**: 適切なアイコンが特定できなければ **埋め込まず省略** する（無関係な画像や推測 URL で代替しない。捏造禁止）

### 4-3. 英語ソース主体のトピック

- **本文は日本語標準語**
- **原文引用はそのまま（英語）**、引用ブロック直後に日本語要約を添える
- 専門用語はカナ表記 + 英語併記（例: プロファイリング（profiling））
- 参照元一覧のタイトルは原題（英語）

## 5. プレビュー

ファイル書き出し完了後、**絶対パスを表示** してから `~/bin/md-open "{絶対パス}"` を 1 回のみ実行（上位 CLAUDE.md の規定に準拠）。

```bash
~/bin/md-open "/absolute/path/to/report.md"
```

小刻みな編集の途中では実行しない。最終成果物としてユーザーに提示する時点のみ。

## 出力形式

```markdown
# 調査レポート: {トピック}

![{トピック名} のアイコン/シンボル](画像URL)

調査日: YYYY-MM-DD

## 概要

{トピックの概要を簡潔にまとめる（200〜400 字）}

## 再調査時のプロンプト案

{再調査が必要な場合のプロンプト例を日本語文字ベースで最大 400 字程度で記載}

## まとめ

{調査結果の総括。決定的な知見が得られなかった場合はその旨を明記}

## 調査結果

### {サブトピック1}

{調査内容}

> {参考にした箇所の引用（原文まま）}
>
> — 出典: [タイトル](URL)

### {サブトピック2}

{調査内容}

> {参考にした箇所の引用}
>
> — 出典: [タイトル](URL)

## 参照元一覧

| タイトル | URL | 参照日 |
|---------|-----|--------|
| {タイトル1} | {URL1} | YYYY-MM-DD |
| {タイトル2} | {URL2} | YYYY-MM-DD |
```

- サブトピックは **3〜8 個** を目安（8 を超える場合はカテゴリで集約してネストすること）
- 引用ブロックの形式は厳守（`> 引用` 空行 `> — 出典: [タイトル](URL)`）
- 同じ URL を複数サブトピックで引用しても良いが、参照元一覧では 1 行にまとめる
- 参照元一覧は **本文で実際に引用または内容参照したソースのみ** を列挙する（読んだが使わなかったものは載せない）

## 重要なルール

1. **参照 URL は必ず明記する** — 情報の出典となる URL を全て記載
2. **引用は明確に** — 参考にした箇所は `>` でブロック引用し、出典を添える
3. **複数ソース必須** — 最低 3 ソース（1 節「複数ソース必須」参照）
4. **捏造禁止** — 検索で確認できない情報を推測で補わない
5. **調査日を記録** — レポートに調査日を必ず含める
6. **画像は引用ポリシーに従う** — 原則は外部リンク埋め込み、ダウンロード保存は特別な場合のみ
7. **先頭にトピックアイコン** — タイトル直下にトピックを象徴するアイコン/シンボル画像を 1 枚埋め込む（見つからなければ省略。4-2 節参照）
